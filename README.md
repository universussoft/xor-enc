# XOR ENC — One-Time Pad File Encryptor

> **Perfect security. No server. No password. No trace.**

A single-file HTML application that encrypts and decrypts files using the **One-Time Pad (OTP)** cipher — the only encryption method proven to be mathematically unbreakable.

---

## How It Works

OTP encryption is simple and provably secure:

```
ENCRYPT:   original_file  XOR  random_pad  =  file.xenc
DECRYPT:   file.xenc      XOR  random_pad  =  original_file
```

When you encrypt a file, the app generates a **cryptographically random pad** (using `crypto.getRandomValues()`) that is **exactly the same size** as your file. The pad is the key. There is no password to remember or forget.

---

## Usage

### Encrypt

1. Open `xor-enc.html` in any modern browser
2. Drag your file(s) into the drop zone
3. Click **Encrypt**
4. Download **both output files**:
   - `file.xenc` — your encrypted data (safe to share)
   - `file.xenc.pad` — your secret key (never share)

### Decrypt

1. Open `xor-enc.html`
2. Drag the `.xenc` file into the drop zone
3. Drag the corresponding `.pad` file into the pad drop zone
4. Click **Decrypt**
5. Download the recovered original file

---

## ⚠ Critical Rules

| Rule | Why |
|------|-----|
| **Save both files** | You need both `.xenc` and `.pad` to decrypt |
| **Never reuse a pad** | Reusing a pad for two files destroys all security — an attacker can XOR the two ciphertexts and recover the data |
| **Keep the .pad secret** | The `.xenc` file is safe to share publicly. The `.pad` must stay private |
| **The pad is not stored in the encrypted file** | If you lose the `.pad`, the data is **permanently unrecoverable** |

---

## Security Properties

- **Perfect secrecy** — proven by Claude Shannon (1949). A ciphertext encrypted with a true OTP reveals zero information about the plaintext, even with unlimited computing power
- **No password** — security comes from the randomness of the pad, not a human-chosen passphrase
- **No key derivation** — no PBKDF2, no bcrypt, no stretching needed — the pad is already maximum entropy
- **Quantum-resistant** — since OTP security is information-theoretic (not computational), quantum computers offer no advantage to an attacker

---

## Why We Don't Use OTP for Everything

| Feature | OTP | AES-256 |
|---------|-----|---------|
| Security | Perfect | Very high |
| Key size | = file size | 32 bytes |
| Key management | Impossible at scale | Practical |
| Speed | Fast | Fast |
| Distribution | Requires secure channel | Diffie-Hellman |

OTP is the theoretical "Holy Grail" of cryptography, but impractical for general use because the key must be as large as the message and can never be reused. For most purposes, AES-256 is the pragmatic choice.

---

## Historical Note

The OTP was used on the **Moscow–Washington Hotline (1963)** during the Cold War:
- Keys generated by roulette machines — true physical randomness
- Delivered by diplomatic courier between the two superpowers
- Every message used a unique key, immediately destroyed after use

It remains the only communication system both sides considered absolutely secure.

## Running Locally

Because browsers block some file operations on `file://` URLs, serve the file over HTTP:

```bash
# Python 3
python -m http.server 8080
# then open http://localhost:8080/xor-enc.html
```

```bash
# Node.js
npx serve .
```

Or use the **Live Server** extension in VS Code.
